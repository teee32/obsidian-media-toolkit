As-of Date: 2026-03-05

我会很直说：这个仓库目前更像“半改名的 Obsidian 图片管理插件原型”，而不是一个能让人放心装进日常库里的“Media Manager”。最大的问题不是功能少，而是身份/工程/实现一致性都没收口，导致“看起来能用”和“实际上可维护、可发布”之间差得有点远。 [1]  

⸻

结论先给：现在最致命的 5 个硬伤（按优先级）

1) 项目身份混乱到影响可发布性：repo 叫 media-manager，但核心元数据还是 image-manager
    •    manifest.json 的 id 是 obsidian-image-manager，name 是 Image Manager，描述也是 “Manage images …”。[2]  
    •    package.json 的 name 也还是 obsidian-image-manager，描述同样是 image management plugin。[3]  
    •    README 却在讲 Obsidian Media Manager。[4]  

这不是“命名瑕疵”，而是插件生态里会直接造成冲突/混淆的级别：你就算把代码写完了，上架/分发/用户安装都会被这些元数据拖后腿。[2][3][4]  

⸻

2) 你还撞上了“现存同名项目”的雷：obsidian-image-manager 这名字在 GitHub 上已经有人用

GitHub 上已经存在 davidvkimball/obsidian-image-manager，项目名就叫 Image Manager for Obsidian，并且已经有版本迭代与 releases（截至 2026-03-03 还有 Latest）。[5]  

这意味着：你现在这个仓库沿用 obsidian-image-manager 作为 id/name 的做法，非常容易造成品牌/搜索/生态层面的碰撞（即使你不是抄袭，用户也会以为你是那个项目的 fork/山寨/镜像）。[2][5]  

我不会直接说你“抄”，但我会说：你给人的观感就是“像没改干净的 fork”，因为 repo 里连架构文档文件名都还带 OBSIDIAN-IMAGE-MANAGER-...。[1]  

⸻

3) README 可信度偏低：Issues 链接还是占位符

README 的反馈链接写的是 https://github.com/your-repo/obsidian-media-manager/issues 这种模板占位符。[4]  

这类细节会让人立刻判断：作者没有把“对外可用”当成目标，更像是练手/临时放上来。[4]  

⸻

4) “图片对齐”功能目前看起来没接通，甚至有明显实现自相矛盾点

你确实放了 utils/imageAlignment.ts，但它现在有几个非常要命的问题：
    •    同一个文件里同时 export type ImageAlignment = ... 和 export class ImageAlignment 同名。[6]  
    •    applyAlignment() 先用正则尝试移除 { align : ... }（冒号风格），但实际新增的是 {align=center}（等号风格）。[6]  
    •    getAlignment() 又只识别 { align : ... }（冒号风格），也就是说：你自己写进去的对齐标记，自己又读不出来。[6]  
    •    更关键的是：主插件 main.ts 里注册的命令只有“图片库 / 查找未引用图片”，没看到任何“对齐”命令入口。[7]  
    •    设置页也只有 folder / thumbnail / sort 等选项，没有 alignment 相关设置。[8]  

综合起来，这个“图片对齐”现在更像是丢在仓库里的未完成实验文件，而不是用户可用的功能。[6][7][8]  

⸻

5) “未引用图片”扫描逻辑在真实 Obsidian 语法里会大量误判，然后你还提供了“删除全部”

扫描引用的核心是两条 regex：
    •    Wiki 链接：[[...png]]（不支持 |alias 的那类形式）[9]  
    •    Markdown 图片：![](...png)（并且只把路径最后的文件名塞进集合）[9]  
对比时却用 file.name（只有文件名，不带目录）。[7]  

这会导致一些很现实的坑（都属于“你迟早会遇到”的那种）：
    •    你的 vault 里如果存在同名图片（不同文件夹），按文件名匹配会误判引用关系。[7][9]  
    •    WikiLink 如果写成 [[attachments/img.png]]（带路径），你会把 attachments/img.png 放进 referenced set，但对比时拿 img.png 去比，直接判成未引用。[7][9]  
    •    [[img.png|300]] / [[img.png|alias]] 这类在 Obsidian 里很常见的写法，你这条 wiki regex 根本不吃，会漏掉引用。[9]  

然后在 UI 里你提供：
    •    单张删除：vault.delete(image.file)。[10]  
    •    删除全部：确认一次后循环删完，再重新扫描。[10]  

这就是典型的“误判 + 强破坏性操作”组合：很容易让用户把其实仍被引用的附件删掉（尤其是带路径/别名/同名的情况）。[9][10]

功能层面：它到底做了什么（实话实说）

✅ 已实现且能看出意图的部分
    •    图片库视图（Image Library view）：扫描图片扩展名，按设置的 folder 前缀过滤，支持按名称/日期/大小排序，右键菜单可复制路径、复制 [[file]]、打开原文件。[11]  
    •    未引用图片视图：扫描后展示列表、统计总大小，提供“在笔记中查找 / 复制路径 / 删除”。[12]  
    •    设置页：folder、缩略图大小、默认排序、是否显示信息、自动刷新等（偏基础）。[8]  

这些属于“Obsidian 插件入门作品里比较常见、也确实有用”的功能块。[8][11][12]  

⸻

代码质量与 UX：不留情面讲讲“糙”的地方

A) 图片库 header 计数是逻辑 bug：你在拿到图片列表之前就渲染了“共 N 张”

refreshImages() 里先 renderHeader()，之后才去读取图片文件并给 this.images 赋值。[11]  
但 renderHeader() 的统计直接用 this.images.length。[11]  

所以你很可能会看到“共 0 张图片”这种一眼劝退的小问题（除非刚好沿用了上一次的 images 状态）。[11]  

⸻

B) “在笔记中查找”其实是全库暴力字符串 contains，而且找到行号也没用上

openImageInNotes() 会把所有 md 文件读出来，按行 line.includes(imageFile.name) 扫过去。[7]  
它记录了 line，但最后只是 openLinkText(file.name, file.path, true)，并没有跳转到具体行。[7]  

这会带来两个体验问题：
    •    大库里会慢（IO + split + 全扫描）。[7]  
    •    精度差：只要正文里提到同名字符串也会命中，不一定是图片引用。[7]  

⸻

C) 删除交互很“野”：直接用浏览器 confirm()，且批量删除没进度/没兜底

单张删除和批量删除都走 confirm(...)。[10]  
批量删除是同步循环删，并且错误只统计数量，没有逐项可追溯的恢复路径。[10]  

在 Obsidian 生态里，这种删除 UX 一般会被认为“不够专业 / 不够安全”。[10]  

⸻

“它配叫 Media Manager 吗？”——目前不配

你现在只处理图片扩展名列表（png/jpg/gif/webp/svg/bmp），没有看到对音视频、PDF 等“media”常见附件的处理逻辑。[7]  
所以“media-manager”这个名字在当前实现下属于过度营销/命名虚高。[7]  

⸻

上架与生态：README 说能在插件市场装，但证据不支持

Obsidian 官方 community plugin registry（obsidianmd/obsidian-releases/community-plugins.json）里搜索不到 obsidian-media-manager。[Single Source][13]  
同一个 registry 里也搜索不到 obsidian-image-manager。[Single Source][14]  

所以 README 那段“从插件市场安装（推荐）”目前更像是套模板而不是事实陈述。[4][13][14]
